## Codebase Patterns
- `NotifyConfig.on_failure` and `on_success` are `Vec<String>` with `string_or_vec` serde deserializer â€” accepts both single string and array in YAML/TOML
- `resolve_notifier(url)` in `resolve.rs` is the unified entry point for parsing notification URL strings into `Box<dyn Notifier>` â€” use this instead of constructing notifiers directly
- The legacy `NotifyCommand`/`resolve_notify_command`/`run_notify` shell-out code has been removed â€” all notification delivery now goes through trait-based `Notifier` implementations
- `send_notifications()` in `executor.rs` is the top-level function for post-workflow notifications â€” it builds notifiers from config, constructs a `Notification`, and fans out via `MultiNotifier`
- `build_notification()` creates a `Notification` struct from `RunLog` â€” use this when you need to build notifications outside the executor
- All `pub mod` declarations are in `src/core/mod.rs` â€” the `pub mod notify;` line works for both file and directory modules, no change needed
- Existing executor uses `crate::core::notify::NotifyCommand` and `crate::core::notify::resolve_notify_command` â€” backward compat re-exports in mod.rs keep these working
- Tests in this crate are inline (`#[cfg(test)] mod tests`) not in separate test files
- Integration tests are in `tests/cli_tests.rs` using `assert_cmd`
- Feature flags go in `[features]` section BEFORE `[dependencies]` in Cargo.toml â€” each notification backend has its own feature gating its HTTP/SMTP dep
- `ureq` (sync HTTP) is used instead of `reqwest` to avoid pulling in tokio â€” all notification backends that do HTTP share the `ureq` optional dep
- `lettre` is used for SMTP email â€” gated behind `email` feature
- Notifier module pattern: `#[cfg(feature = "X")] pub mod X;` in mod.rs, struct with `new()`, private `build_payload()` for testable JSON, `Notifier` trait impl with `ureq::post`
- Sort HashMap-derived fields alphabetically in `build_payload()` for deterministic test assertions
- lettre 0.11: use `message.formatted()` for test assertions (body field is private); `SmtpTransport::builder_dangerous()` returns builder directly (not Result)
- `NotifyConfig.channels` is `Vec<NotifyChannel>` with `target` (URL string) and `on` (Vec<Severity>) â€” channels take precedence over legacy `on_failure`/`on_success` within the same config level
- `resolve_targets()` in `executor.rs` is the unified target resolution function â€” it **merges** workflow + global targets by default (dedup by URL); set `notify_override: true` on workflow config to fully replace global
- `collect_targets()` is the per-config helper that resolves channels vs legacy for a single `NotifyConfig` â€” used by `resolve_targets()` for both workflow and global levels
- `Severity` has `#[serde(rename_all = "lowercase")]` so it deserializes from "success"/"failure"/"warning"/"info" strings in YAML/TOML
- `RetryConfig` in `notify/retry.rs` provides exponential backoff; `send_with_retry()` is the per-notifier retry function â€” use `resolve_retry_config()` in executor to get the effective config from workflow/global
- When adding new fields to `NotifyConfig`, use `..Default::default()` in existing tests to avoid breakage â€” don't list all fields explicitly
- `RateLimiter::with_configs()` merges user overrides on top of built-in defaults (Discord 30/min, Telegram 30/sec) â€” use this constructor when building from config
